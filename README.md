# ai-keiei-kaigi — 仮想経営会議

中小企業の経営者の相談に対して、**4人の専門AI（財務・営業・現場・リスク）が会議を行い、
議長AIが「提言書」1枚に統合して返す** マルチエージェントシステムです。

```
経営者の相談
   │
   ▼
┌─────────────── AI経営会議 ───────────────┐
│ Round 1  財務・営業・現場・リスクが独立に意見       │
│ Round 2  互いの意見をレビュー（見落とし・過大評価の指摘）│
│ Round 3  議長が提言書に統合                  │
└──────────────────────────────────────┘
   │
   ▼
提言書: ①事実情報の整理 ②経営者への質問（不足情報）
        ③論点 ④次に考えるべきこと ⑤相談すべき先
```

## 設計のポイント

1. **AIは結論を押し付けない。** 提言書は「判断材料の整理」に徹し、
   事実・仮定・不足情報を厳密に分離する。不足情報はAIが補完せず、
   「経営者への質問」として返す。
2. **相互レビュー（Round 2）を挟む。** 独立意見をそのまま束ねると各AIの
   楽観・悲観が提言に残るため、批評の段を挟んで根拠の弱い主張を落とす。
   （批評エージェントの導入でタスク成功率が 27%→89% に改善した
   自作マルチエージェントでの知見を踏襲）
3. **役割の壁を明示する。** 各専門AIに「見る指標」と「やってはいけないこと
   （他の担当の領分を勝手に仮定しない）」を定義し、意見の重複と越権を防ぐ。
4. **難しい相談は人につなぐ。** 提言書の最終節は必ず「相談すべき先」
   （税理士・金融機関担当者など、誰に・何を持って行くか）で終わる。

## 実行方法

```bash
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env   # GEMINI_API_KEY を設定（無料枠で動作）

# サンプルケースで実行
python main.py --input examples/case1_設備投資/相談.md --outdir examples/case1_設備投資

# 相談を直接入力
python main.py --consult "配送トラックを1台増やすべきか迷っている。..." --title "トラック増車"
```

出力は `提言書.md`（経営者向け1枚）と `会議ログ.md`（全AIの発言記録）。

## 評価

`eval/rubric.md` の8項目チェックリストで各ケースの提言書を評価しています
（事実と推測の分離 / 不足情報の質問化 / 論点の明示 / 相談先の具体性 など）。

## 技術構成

- Python 3.12 / Gemini 2.5 Flash（無料枠）
- `llm_client.py`: プロバイダ抽象化（Gemini/OpenAI/Claude 切替、
  複数キー自動ローテーション、429/503 での自動リトライ・フォールバック）
- 外部フレームワーク不使用（エージェント間の情報の流れを全て自前で制御）

## 制限事項（正直に）

- 専門AIの知識は LLM の一般知識であり、業種固有の相場観は保証しない
- 提言書は判断材料の整理であって、財務・法務アドバイスではない
- 現状は1ターンの会議のみ（追加質問への回答を受けて再会議する機能は未実装）
